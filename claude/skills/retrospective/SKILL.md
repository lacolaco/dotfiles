# Retrospective

コミット・PR作成前に実施する。

コルブの経験学習サイクルの4段階を順に実行する。Phase 3（概念化）が最重要。

## Phase 1: Concrete Experience

事実のみ記録する。解釈・評価を混ぜない。

- **Goal**: 要件 / 完了した成果物 / 未完了・ブロッカー
- **Efficiency**: 実行したアクション / やり直し箇所 / 遭遇した障害
- **Work Units**: 複数ステップで構成された作業単位（例: PR作成→CI監視→マージ、依存追加→lock更新→検証）。成功/失敗を問わず列挙しろ
- **User Satisfaction**: ユーザーの反応 / 不満を示した場面 / 満足を示した場面

## Phase 2: Reflective Observation

Phase 1の事実からパターンを見出す。「何が起きたか」ではなく「なぜそうなったか」を問う。

失敗・ギャップの省察:

- 成功と失敗の構造的な差は何か
- ユーザーの不満は何の欠如を示しているか
- やり直しの原因はどの判断時点に遡るか
- 暗黙知に依存していなかったか（CLAUDE.mdに書かれていない前提）
- 既存ルール（CLAUDE.md、スキル記述）自体が問題を誘発していないか。字義通り解釈して暴走するなら、ルール側の表現・粒度に欠陥がある。自己の解釈だけを犯人にして止めるな

作業パターンの省察:

- Phase 1の Work Units のうち、今後も繰り返し発生するものはどれか
- 同じ作業単位が過去のセッションにもあったか
- 既存スキルで包摂されているか、抜け漏れはどこか
- 複数ステップが暗黙の順序・条件で連結されていなかったか

## Phase 3: Abstract Conceptualization

省察から以下を抽出する。最重要ステップ。

### 再利用可能な原則

原則の品質基準:

- 今回固有でなく、同類の問題全般に適用できる
- 「Xを検証する」でなく「検証なしに前提を置かない」のレベルで抽象化されている
- 原則を読んだだけで未経験の状況でも正しい判断ができる

悪い例: 「Gemini APIの `oneOf` サポート状況を事前に検証する」
良い例: 「二次情報を因果推論の根拠にしない。実機検証か一次ソースで裏付ける」

### 継続的に発生する作業パターン

Phase 2で識別した作業単位のうち、以下を満たすものを抽出する:

- 複数ステップで構成される
- 今後も繰り返し発生する
- 既存スキルで包摂されていない、または既存スキルの拡張で対応可能

悪い例: 「今回編集した特定ファイルの手順」（局所的すぎる）
良い例: 「PRマージ後のポストマージワークフロー（ブランチ削除→retrospective→handover更新）」

## Phase 4: Active Experimentation

Phase 3の成果を以下のフローで処理する。**必ず Step 1 から順に判定しろ。** 各ステップで「この手段で対処可能か」を問い、可能なら実施してフロー終了。不可なら次ステップへ。上位をスキップして下位に飛ぶな。

### Step 1: 不要化できるか

ルール・作業そのものが発生しない仕組みにできないか問う。アーキテクチャ変更、自動化による手順の省略、問題が構造的に起こらない設計。

- 可能 → 実施してフロー終了
- 不可 → Step 2 へ。不可と判断した根拠を記録しろ

### Step 2: 決定論的ガードレールで強制できるか

機械が強制・検知できないか問う。lint, typecheck, CI, pre-commit hooks, `settings.json` hooks等。

- 可能 → 実施してフロー終了
- 不可 → Step 3 へ。不可と判断した根拠を記録しろ

### Step 3: スキル化できるか

継続的に発生する作業パターンなら `SKILL.md` に手順として分離する。既存スキルで包摂できるなら拡張。

配置レイヤー:

- 普遍的なワークフロー・全環境共通の作業パターン → グローバル `~/.claude/skills/<name>/SKILL.md`
- workspace固有の作業パターン → workspace `<workspace>/.claude/skills/<name>/SKILL.md`

判定:

- 対象が作業パターン → 必ずここで処理（例外なし）。フロー終了
- 対象が原則で、スキル化で表現不可 → Step 4 へ

### Step 4: エージェントプロンプトで表現できるか

特定エージェントの挙動として定義できないか問う。

配置レイヤー:

- 普遍的・全環境共通のエージェント挙動 → グローバル `~/.claude/agents/<name>.md`
- workspace固有のエージェント挙動 → workspace `<workspace>/.claude/agents/<name>.md`

判定:

- 可能 → 実施してフロー終了
- 不可 → Step 5 へ。不可と判断した根拠を記録しろ

### Step 5: workspace CLAUDE.md追記（最終手段）

Step 1〜4すべてで不可と判定された場合のみ追記する。対象はそのworkspace固有の具体的な作業、問題解決、ワークフロー、ドメイン知識。

**CLAUDE.mdが最悪の手段である理由: モジュール化されず、関心の分離ができないから。** Step 1〜4はいずれもモジュール境界を持つ（設計単位、ガードレール単位、スキル単位、エージェント単位）。CLAUDE.mdは単一ファイルに全責務が積み上がり、肥大化・文脈汚染・責務の混在を引き起こす。

### グローバルCLAUDE.mdの扱い

グローバル `~/.claude/CLAUDE.md` が扱うのはAI自身のパーソナリティ構築レイヤ（Principle、行動規範、思考様式、人格形成）のみ。具体的な作業・問題解決・ワークフローをグローバルに書くな。

Phase 4の対象がパーソナリティレイヤに該当する場合のみ、例外的にグローバルへ追記する。それ以外はすべてworkspaceローカルに書け。

### 規律（CLAUDE.md追記共通）

- 既存ルールで包摂できる内容の問題が再発したなら、既存ルールの表現・配置・優先度を改訂しろ。機能していないルールを放置したまま似たルールを重ねるな
- ルールを足すだけで問題に対処した気になるな。再発は既存ルールの不備の証拠であり、原因（表現不足・配置ミス・優先度不足）に向き合え
- 1原則1行。具体例は最小限
- 類似項目は統合しろ。肥大化させない
- レイヤーを混ぜるな。具体的作業をグローバルに書くな

## 提出

Phase 1〜4の結果をユーザーに提示し、改善を反映する。反映後、未コミットの変更があればコミット→pushまで自律実行する。
