# Retrospective

コミット・PR作成前に実施する。

エージェントのタスク実行は 6 段階で進む:

**入力 → 解釈 → 計画 → 行動 → 検査 → 出力**

振り返りは下流 → 上流に **Problem をボトムアップで吸い上げ**、上流 → 下流に **Try をトップダウンで解決**する。**上流で塞いだ穴は下流で塞ぎ直さない**。多段防御は上流対策の確実性が低い場合のみ例外的に張る。

## 6段階の定義

- **入力**: 指示・context・skill・CLAUDE.md・ツール description の受領、および Read/Grep/Glob/WebFetch 等による能動収集
- **解釈**: 入力の意味・意図・前提の読み取り
- **計画**: タスク分解・順序決定・ツール選定・スコープ確定
- **行動**: ツール呼び出し・ファイル変更・コマンド実行
- **検査**: 結果の検証・合格判定
- **出力**: ユーザーへの報告・永続化判断

## Phase 1: 事実記録

6段階それぞれで起きた事実を時系列で記録する。解釈・評価を混ぜない。

- **入力**: 受領した指示 / 収集した情報とその一次性
- **解釈**: 読み取った意図 / 置いた前提
- **計画**: 分解した手順 / 選定したツール / 決めたスコープ
- **行動**: 実行した操作 / 発生した副作用 / やり直し箇所
- **検査**: 実施した検証 / 採用した合格基準
- **出力**: 報告内容 / 永続化の有無 / ユーザー反応（不満・満足）

## Phase 2: Problem のボトムアップ洗い出し（出力 → 入力）

下流から上流へ順に、各段階で Problem を洗い出す。各段階で必ず問え:

1. この段階で発生した Problem は何か
2. 下流の Problem を、この段階で検出・修正できたはずか
3. 真因はこの段階か、さらに上流か

順序（逆引き）:

1. **出力**: 報告の過不足 / 永続化抜け / ユーザー不満の具体
2. **検査**: 検証省略 / 合格基準の誤り / 検査方法の不適
3. **行動**: ツール誤用 / 引数誤り / 副作用見落とし / 環境認識ミス
4. **計画**: 分解粒度の誤り / 順序ミス / スコープ逸脱
5. **解釈**: 意図の誤読 / 前提の取り違え / 矛盾の見落とし
6. **入力**: 情報不足 / 誤情報混入 / 収集不足 / 一次ソース未参照

### 既存ルール自体が Problem を誘発していないか

CLAUDE.md やスキルの記述が字義通り解釈を招き暴走を引き起こすなら、ルール側の表現・粒度に欠陥がある。自己の解釈だけを犯人にして止めるな。

### 入力の誤りは不足と並ぶ独立要因

「情報不足」と「誤情報混入」は別物。対策も別。誤情報の源（古い skill、陳腐化した CLAUDE.md、ツール description の実装乖離、二次情報の一次扱い、記憶のハルシネーション、前セッション要約の歪み）を特定しろ。

## Phase 3: Keep の抽出

各段階で維持すべき成功パターンを抽出する。

- どの段階で何が機能したか
- 再利用可能な形に抽象化できるか

品質基準:

- 今回固有でなく、同類の状況に適用できる
- 「X を検証した」でなく「検証なしに前提を置かなかった」のレベル

## Phase 4: Try のトップダウン展開（入力 → 出力）

Phase 2 で特定した真因に対し、上流から順に Try を立てる。**上流で塞いだ穴は下流で塞がない**。

各段階で Try を立てる前に問え:

- 上流ですでに真因を絶ったなら、この段階の対策は不要
- 多段防御を張るのは、上流対策の確実性が低い場合のみ

順序（順行）:

1. **入力**: 情報源の優先順位定義 / 能動収集の規律 / 陳腐化した skill・CLAUDE.md・ツール description の修正
2. **解釈**: 確認条件の追加 / 前提の明示化 / 矛盾検出の規律
3. **計画**: 計画テンプレ化 / 分解粒度の規律 / スコープ定義
4. **行動**: ツール選択基準 / 操作パターンの skill 化 / 副作用の事前列挙
5. **検査**: 検査項目の明文化 / 合格基準の定義
6. **出力**: 報告形式 / 永続化判断の規律

### 継続的に発生する作業パターン

Phase 2 の Problem のうち以下を満たすものは skill 化候補:

- 複数ステップで構成される
- 今後も繰り返し発生する
- 既存 skill で包摂されていない、または拡張で対応可能

悪い例: 「今回編集した特定ファイルの手順」（局所的すぎる）
良い例: 「PR マージ後のポストマージワークフロー（ブランチ削除→retrospective→handover更新）」

### 原則の品質基準

悪い例: 「Gemini API の `oneOf` サポート状況を事前に検証する」
良い例: 「二次情報を因果推論の根拠にしない。実機検証か一次ソースで裏付ける」

## Phase 5: 改善策の実装フロー

Phase 4 の各 Try を以下のフローで処理する。**必ず Step 1 から順に判定しろ**。各ステップで「この手段で対処可能か」を問い、可能なら実施してフロー終了。不可なら次ステップへ。上位をスキップして下位に飛ぶな。

### Step 1: 不要化できるか

ルール・作業そのものが発生しない仕組みにできないか問う。アーキテクチャ変更、自動化による手順の省略、問題が構造的に起こらない設計。

- 可能 → 実施してフロー終了
- 不可 → Step 2 へ。不可と判断した根拠を記録しろ

### Step 2: 決定論的ガードレールで強制できるか

機械が強制・検知できないか問う。lint, typecheck, CI, pre-commit hooks, `settings.json` hooks等。

- 可能 → 実施してフロー終了
- 不可 → Step 3 へ。不可と判断した根拠を記録しろ

### Step 3: スキル化できるか

継続的に発生する作業パターンなら `SKILL.md` に手順として分離する。既存スキルで包摂できるなら拡張。

配置レイヤー:

- 普遍的なワークフロー・全環境共通の作業パターン → グローバル `~/.claude/skills/<name>/SKILL.md`
- workspace固有の作業パターン → workspace `<workspace>/.claude/skills/<name>/SKILL.md`

判定:

- 対象が作業パターン → 必ずここで処理（例外なし）。フロー終了
- 対象が原則で、スキル化で表現不可 → Step 4 へ

### Step 4: エージェントプロンプトで表現できるか

特定エージェントの挙動として定義できないか問う。

配置レイヤー:

- 普遍的・全環境共通のエージェント挙動 → グローバル `~/.claude/agents/<name>.md`
- workspace固有のエージェント挙動 → workspace `<workspace>/.claude/agents/<name>.md`

判定:

- 可能 → 実施してフロー終了
- 不可 → Step 5 へ。不可と判断した根拠を記録しろ

### Step 5: workspace CLAUDE.md追記（最終手段）

Step 1〜4すべてで不可と判定された場合のみ追記する。対象はそのworkspace固有の具体的な作業、問題解決、ワークフロー、ドメイン知識。

**CLAUDE.mdが最悪の手段である理由: モジュール化されず、関心の分離ができないから。** Step 1〜4はいずれもモジュール境界を持つ（設計単位、ガードレール単位、スキル単位、エージェント単位）。CLAUDE.mdは単一ファイルに全責務が積み上がり、肥大化・文脈汚染・責務の混在を引き起こす。

### グローバル CLAUDE.md の扱い

**retrospective の成果として、グローバル `~/.claude/CLAUDE.md` を改変するな**。追加・削除・修正のいずれも禁止。Phase 4 の対象がパーソナリティレイヤに該当すると判定した場合でも、retrospective からグローバルへは書き込まない。グローバル CLAUDE.md の編集はユーザーが明示的に指示したときのみ実施する独立タスクであり、retrospective のスコープ外。

retrospective の成果はすべて workspace ローカルに書け。配置先は workspace CLAUDE.md / skill / agent のいずれか。

### 規律（CLAUDE.md 追記共通）

- 既存ルールで包摂できる内容の問題が再発したなら、既存ルールの表現・配置・優先度を改訂しろ。機能していないルールを放置したまま似たルールを重ねるな
- ルールを足すだけで問題に対処した気になるな。再発は既存ルールの不備の証拠であり、原因（表現不足・配置ミス・優先度不足）に向き合え
- 1原則1行。具体例は最小限
- 類似項目は統合しろ。肥大化させない
- レイヤーを混ぜるな。具体的作業をグローバルに書くな

### Step 6: 新ルールをセッション内既存成果物に遡及適用する

Step 1〜5 で新しいルール / skill / ガードレールを制定したら、そのセッション内で既に生成した成果物（issue / MR description / コメント / ドキュメント / コード / コミットメッセージ）を遡って新ルールでチェックする。

- 制定と適用は別工程。作成しただけで直近のアウトプットが改善されるわけではない
- 新ルールに違反する箇所を検出したら、元の成果物を更新する（GitLab / GitHub / ファイル）
- 違反なしを確認した場合も「確認した」旨を retrospective 提出に含める

直前に作ったルールに違反する成果物がそのまま残っていれば、retrospective の成果は空回りする。

## 提出

Phase 1〜5 の結果をユーザーに提示し、改善を反映する。反映後、未コミットの変更があればコミット→push まで自律実行する。
